# OwlAlpha 后端设计

## 1. 设计目标

后端使用 GoFrame 作为基础框架，承担统一 API 网关、业务编排、任务调度、数据管理和 AI 分析调用等职责。设计目标是在保持单体应用易部署特性的同时，提供清晰的分层结构，为未来功能扩展与模块拆分奠定基础。

## 2. 技术基线

- Go 1.23+
- GoFrame 2.x
- PostgreSQL
- Redis
- Docker Compose 部署

## 3. 目标目录结构

```text
backend/
├── api/                    # API 定义
├── hack/                   # 开发辅助脚本
├── internal/
│   ├── controller/         # HTTP 控制器
│   ├── logic/              # 业务逻辑实现
│   ├── service/            # 服务接口定义
│   ├── dao/                # 数据访问对象
│   ├── model/              # 数据模型与实体
│   ├── jobs/               # 定时任务与异步任务
│   ├── middleware/         # 鉴权、日志、恢复等中间件
│   ├── consts/             # 常量定义
│   └── packed/             # 打包资源和内置配置
├── manifest/
│   ├── config/             # 配置模板
│   └── sql/                # 初始化 SQL / migration
├── resource/               # 静态资源与模板
└── main.go                 # 启动入口
```

## 4. 分层约束

### 4.1 controller

- 负责请求解析、参数校验和响应返回。
- 不承载复杂业务逻辑。
- 调用 `service` 层暴露的接口完成业务处理。

### 4.2 service

- 定义业务能力接口。
- 提供依赖注入边界。
- 屏蔽具体 `logic` 实现细节。

### 4.3 logic

- 承担核心业务编排。
- 调用数据访问层、外部数据源和模型接入层。
- 负责事务、状态流转、规则判断和结果组装。

### 4.4 dao / model

- 负责数据库访问和实体映射。
- 避免在数据访问层写入复杂跨领域编排逻辑。

### 4.5 jobs

- 管理定时任务、批量任务、重试和执行状态更新。
- 与 `analysis`、`report`、`stock` 等业务模块协作。

## 5. 领域模块设计

### 5.1 auth 模块

职责：

- 本地账号登录。
- JWT 或 Session 管理。
- 权限校验和中间件接入。

典型接口：

- `POST /api/v1/auth/login`
- `POST /api/v1/auth/logout`
- `GET /api/v1/auth/me`

### 5.2 system 模块

职责：

- 系统基础配置管理。
- 模型提供方配置管理。
- 数据源配置管理。
- 健康检查与运行状态查询。

### 5.3 stock 模块

职责：

- 股票基础信息维护。
- 股票池、分组、自选列表管理。
- 分析启停状态控制。

典型实体：

- `stocks`
- `stock_groups`
- `stock_pool_items`

### 5.4 market 模块

职责：

- 获取 A 股行情和 K 线数据。
- 做本地缓存与更新策略控制。
- 输出单股行情摘要和每日报告所需行情上下文。

### 5.5 news 模块

职责：

- 获取新闻资讯。
- 去重、标准化和时效过滤。
- 建立股票与资讯之间的映射关系。

### 5.6 llm 模块

职责：

- 抽象统一模型调用接口。
- 支持 OpenAI 兼容协议。
- 支持自定义 `Base URL`、模型名和 API Key。
- 记录模型名、调用耗时、令牌统计和失败原因。

建议抽象接口：

```go
type Provider interface {
    Chat(ctx context.Context, req *ChatRequest) (*ChatResponse, error)
    Name() string
}
```

### 5.7 analysis 模块

职责：

- 编排行情、新闻和股票基础信息。
- 组装 Prompt 和分析上下文。
- 调用 `llm` 模块生成结果。
- 将结果标准化为统一报告结构。

建议子能力：

- `ContextBuilder`
- `PromptBuilder`
- `AnalysisExecutor`
- `ResultNormalizer`

### 5.8 report 模块

职责：

- 生成分析报告结构。
- 保存历史快照。
- 提供列表、详情和版本追踪。

### 5.9 scheduler 模块

职责：

- 管理手动与定时分析任务。
- 生成批量任务项。
- 记录执行状态、失败重试和运行日志。

## 6. 核心分析链路

后端对单只股票的主链路建议如下：

1. 校验股票是否存在且处于可分析状态。
2. 读取股票基础资料。
3. 拉取最新行情与关键技术摘要。
4. 获取相关新闻并进行清洗和筛选。
5. 组织结构化上下文。
6. 生成 Prompt 并调用目标模型。
7. 对模型结果做标准化与容错处理。
8. 持久化任务记录与报告快照。
9. 返回结果给前端。

## 7. 配置设计

配置建议分为三层：

- 环境变量：数据库、Redis、JWT 密钥、基础运行参数。
- 数据库存储配置：OpenAI 兼容接口配置、数据源、任务策略、系统开关。
- 运行时内存配置：缓存解析后的配置结果，避免频繁查询。

## 8. 错误处理与日志

- 统一错误码和错误响应格式。
- 请求日志、任务日志和模型调用日志分层记录。
- 日志默认避免输出敏感 Key、完整密钥和高风险隐私内容。
- 关键链路记录 trace id，便于排障。

## 9. 数据库设计原则

- 业务表和日志表分离。
- 任务表和报告表分离。
- 原始资讯和分析结果分离。
- 配置变更需可追踪。
- 关键字段建立唯一索引和状态索引。

## 10. 典型 API 分组

- `/api/v1/auth/*`
- `/api/v1/system/*`
- `/api/v1/stocks/*`
- `/api/v1/market/*`
- `/api/v1/news/*`
- `/api/v1/analysis/*`
- `/api/v1/reports/*`
- `/api/v1/tasks/*`

## 11. 可演进方向

- 将任务执行器从主 API 服务中拆分为独立 worker。
- 增加更多模型与数据源 Provider。
- 引入更强的规则引擎和策略模板。
- 增加更丰富的报告中心和权限能力。
