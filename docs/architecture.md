# OwlAlpha 系统架构

## 1. 总体架构说明

OwlAlpha 采用前后端分离的 monorepo 架构，`0.1` 版本将系统划分为后端服务、单一后台前端和部署层三个部分。

系统核心由 GoFrame 后端承担统一 API、业务编排、任务调度和数据管理职责；前端使用单一 `web/` 项目承载后台登录、分析查看和系统配置；部署层通过 Docker Compose 管理 `nginx`、`api`、`db` 和 `redis` 四个核心容器，其中前端静态资源由 `nginx` 统一分发。

## 2. 目标架构

```text
                 +----------------------+
                 |        nginx         |
                 | 后台静态资源 / 反向代理 |
                 +----------+-----------+
                            |
                            | HTTP API
                            |
                 +----------v-----------+
                 |         api          |
                 | GoFrame API Server   |
                 | 调度 / 分析 / 配置    |
                 +----+-----------+-----+
                      |           |
            SQL / ORM |           | Cache / Queue State
                      |           |
            +---------v--+     +--v--------+
            |     db     |     |  redis    |
            +------------+     +-----------+
                      |
                      |
             Files / Snapshots / Logs
                      |
               +------v-------+
               | Local Volume |
               +--------------+

                 +----------------------+
                 |  OpenAI Compatible   |
                 | API via custom URL   |
                 +----------------------+
```

## 3. Monorepo 结构

```text
OwlAlpha/
├── backend/        # GoFrame 后端
├── web/            # 单一后台前端项目
├── deploy/         # Docker Compose / Nginx / env 模板
├── docs/           # 项目文档
└── README.md
```

这种结构有几个目的：

- 统一管理后端、前端和部署配置，降低版本漂移。
- 让分析查看、报告中心和系统配置集中在一个后台应用中。
- 方便 Docker 构建和未来 CI/CD 集成。

## 4. 分层设计

### 4.1 后端层

后端是系统的核心编排层，负责：

- 提供统一 REST API。
- 管理系统配置、股票池、任务和报告。
- 编排行情获取、新闻处理与 LLM 分析。
- 执行定时任务、失败重试和状态追踪。

### 4.2 后台前端

`web/` 面向系统配置与分析使用人员，主要承担：

- 登录与权限控制。
- 股票池、分组、任务、模型和数据源配置。
- 历史报告查看与任务日志追踪。
- 每日报告查看和系统状态管理。

### 4.3 部署与反向代理层

部署层负责：

- 管理容器编排。
- 通过 `nginx` 容器处理统一入口、静态资源分发与反向代理。
- 通过 `api` 容器承载后端服务。
- 通过 `db` 与 `redis` 容器提供数据库和缓存能力。
- 管理环境变量、数据卷和可选服务。

## 5. 核心业务模块

后端建议按领域划分为以下模块：

- `auth`：认证、会话、权限。
- `system`：系统配置、模型配置、数据源配置、健康检查。
- `stock`：股票基础资料、股票池、分组和关注项。
- `market`：行情拉取、K 线缓存、行情摘要。
- `news`：新闻获取、清洗、去重、时效过滤。
- `analysis`：分析编排、Prompt 生成、LLM 调用、结果标准化。
- `report`：报告生成、快照存储、历史查询。
- `scheduler`：定时任务、执行日志、失败重试。
- `llm`：OpenAI 兼容接口接入抽象，支持自定义 `Base URL`。

## 6. 关键数据流

### 6.1 手动单股分析流程

```text
用户发起分析请求
    -> 后端加载股票基础信息
    -> 拉取最新行情与 K 线摘要
    -> 搜集相关资讯并完成过滤
    -> 组装分析上下文与 Prompt
    -> 调用 LLM 生成结构化分析结果
    -> 存储分析报告和任务记录
    -> 返回前端展示
```

### 6.2 定时批量分析流程

```text
调度器扫描启用的股票池
    -> 创建批量分析任务
    -> 按顺序或并发生成任务项
    -> 每只股票执行独立分析流程
    -> 汇总执行结果、耗时与异常信息
    -> 更新任务状态并写入报告历史
```

## 7. 数据存储设计原则

- 结构化业务数据存储于 PostgreSQL。
- 缓存、分布式锁和任务状态辅助信息存储于 Redis。
- 报告快照、运行日志和导出文件存储于本地卷。
- 模型配置和任务执行上下文支持追踪版本。

## 8. 本地优先部署原则

OwlAlpha 的部署设计遵循以下原则：

- 默认不依赖中心化 SaaS 托管服务。
- 数据库和文件卷默认驻留在用户环境中。
- 模型调用仅通过用户配置的 OpenAI 兼容接口完成。
- 所有关键组件均可通过 Docker Compose 独立拉起，默认包含 `nginx`、`api`、`db` 和 `redis` 四个容器。

## 9. 扩展方向

当前架构为后续扩展保留以下空间：

- 增加更多数据源接入。
- 增加更多报告模板与分析策略。
- 拆分分析任务服务，形成更独立的异步架构。
- 增加权限审计、导出与插件化能力。

## 10. 非功能性要求

- 可维护性：模块边界清晰，减少跨层耦合。
- 可扩展性：支持增加更多市场和更多分析模板。
- 可部署性：保证单机和小型服务器可顺利运行。
- 可审计性：记录分析任务、配置变更和报告快照。
- 可恢复性：支持基于数据库和卷数据做备份与恢复。
